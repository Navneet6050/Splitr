## 🔍 3. Cases Where AI Required Correction

### Case 1: Infinite Re-Render Loop in User Identity Sync Hook
* **What the AI generated**: The AI set up an authentication synchronization hook [use-store-user.jsx](file:///c:/Users/manav/OneDrive/Desktop/ai-splitwise-clone/hooks/use-store-user.jsx) that invoked the user storing Server Action on layout mount. However, the custom React mutation hook `useConvexMutation` generated by the AI created a new instance of the `mutate` dispatcher on every render cycles.
* **How it was caught**: During local browser execution, the dev server output logged an endless cascade of duplicate `SELECT` and `INSERT` SQL queries to the `User` table, causing the page to freeze due to stack recursion.
* **How we corrected it**: Wrapped the dispatch function inside `useConvexMutation` in `useCallback` and memoized the query name dependencies inside [use-convex-query.js](file:///c:/Users/manav/OneDrive/Desktop/ai-splitwise-clone/hooks/use-convex-query.js) to stabilize the React dependency array.

### Case 2: Prisma Client Validation Crash on Falsy `"skip"` Query Argument
* **What the AI generated**: When `activeImportId` was falsy (e.g. before an import was initiated), pages queried `useQuery(api.imports.get, activeImportId ? { importId: activeImportId } : "skip")`. The AI's `useConvexQuery` hook intercepted `queryRef === "skip"` but did not check `args === "skip"`, causing it to execute the query against the backend server action with the literal payload `"skip"`.
* **How it was caught**: The server crashed with a `PrismaClientValidationError` in the `get` server action (located in [imports.js](file:///c:/Users/manav/OneDrive/Desktop/ai-splitwise-clone/lib/actions/imports.js)), warning that `id: undefined` was passed to a `findUnique` query.
* **How we corrected it**: Modified [use-convex-query.js](file:///c:/Users/manav/OneDrive/Desktop/ai-splitwise-clone/hooks/use-convex-query.js) line 23 to check:
  `if (queryRef === "skip" || args === "skip")`
  This short-circuited execution early, setting data to `undefined` and preventing the erroneous server call.

### Case 3: Interactive Database Transaction Expiration (`P2028`)
* **What the AI generated**: In [imports.js](file:///c:/Users/manav/OneDrive/Desktop/ai-splitwise-clone/lib/actions/imports.js), the CSV staging and commit actions sequentially iterated through CSV rows using `for (const row of rows)` and ran `await tx.importRow.create(...)`, `await tx.group.update(...)`, and `await tx.expenseSplit.create(...)` queries inside single interactive transactions.
* **How it was caught**: Importing any CSV with 12+ rows threw a transaction expiration error: `Transaction already closed: A query cannot be executed on an expired transaction. The timeout for this transaction was 5000 ms, however 6580 ms passed since the start...`
* **How we corrected it**:
  1. Refactored the staging loop to use Prisma's **`createManyAndReturn`** to stage rows in a single batch query.
  2. Pre-cached membership lookup checks and aggregated `GroupMembership` updates and `ExpenseSplit` records in memory, writing them to PostgreSQL at the end of the transaction via a single **`createMany`** call. This reduced remote WAN roundtrips from 170+ down to 3, speeding up transaction time to under 0.5s.
  3. Configured explicit transaction timeout overrides (`timeout: 60000` for commits) to ensure stable connection allocations under pool load.
